# תסריטי בדיקות קצה-לקצה (E2E) — NatID 360 Control

| שדה | ערך |
|---|---|
| מערכת | NatID 360 Control (Base44) |
| היקף | תהליכי קצה מלאים: ניהול קריאות ← שיוך למוקדן ← שיוך לספק ← טיפול (ספק↔לקוח, מוקדן↔ספק) ← מיקומי ספקים ← התראות ומדדים |
| תפקידים | admin / operator (מוקדן) / vendor (ספק) / agent (טכנאי) |
| שפה/ממשק | עברית, RTL |
| גרסת מסמך | 1.0 |
| תאריך | 09/07/2026 |

> **הערה מקדימה קריטית:** רוב תסריטי הכתיבה (שיוך, שינוי סטטוס, SMS) תלויים בכך שתיקון ההרשאות המערכתי (PR #174 — כל פונקציות ה-backend) פורסם ב-production. ללא פרסום, משתמשים מוזמנים ייחסמו. ראה "תלויות".

## הנחות עבודה

1. **חתימת לקוח** (E2E-VEND-03) — נאכפת בצד הלקוח בלבד; ה-API (updateVendorCall/closeCall) אינו אוכף חתימה. סומן לבדיקת סיכון.
2. **קבוצת שדות "פיננסי" בייצוא מתקדם** (E2E-RPT-04) — נגישה למוקדן, בעוד שדוחות פיננסיים אחרים הם admin-only. ייתכן פער מדיניות — לאמת מול הגדרת המוצר.
3. **מתגי סוגי push** — state מקומי, לא נשמרים בין sessions.
4. **נוסחי SMS סגירה** — placeholder ("נוסח SMS זמני"), טרם הוחלפו בנוסח סופי.
5. **סף טריות מיקום** — התג משתמש ב-10 דקות, ה-nudge ב-5 דקות. ייתכן חוסר עקביות מכוון.
6. **חלק מ-KPIs** (rejection_rate, first_call_resolution, avg_completion_time) — מחזירים null, לא מחושבים עדיין.
7. **מודל הצעה לספק** — שיוך אינו מיידי; נוצרת CallAssignmentAttempt עם תוקף 10 דקות שהספק צריך לאשר.

## סיכונים מרכזיים

1. **מרוצי שיוך (race conditions)** — שני מוקדנים משבצים אותה קריאה במקביל, או ספק מאשר קריאה שכבר נלקחה. יש guards (409) — יש לוודא בעומס.
2. **עקיפת חתימת לקוח דרך API** — אם ה-backend לא אוכף, ספק יכול לסגור ללא חתימה.
3. **דליפת נתונים פיננסיים למוקדן** דרך הייצוא המתקדם.
4. **תלות ב-Twilio/Green API/VAPID** — כשל שקט או 503; SMS לא-חוסם עלול "להיבלע".
5. **סנכרון סטטוסים בין 3 ישויות** (Call/WorkQueue/Case) דרך syncCallStatus — אי-התאמה תשבור תצוגות.
6. **הצעות שפג תוקפן** — קריאה "תקועה" ב"ספק שובץ" אם processStaleAssignments לא רץ.
7. **מיקומים ישנים** — מפות מציגות מיקום לא-עדכני ללא אזהרה בולטת ללקוח.
8. **בידוד נתוני ספק** — ספק חייב לראות רק את הקריאות שלו (getVendorScopedData).

## תלויות לפני תחילת הבדיקות

1. **פרסום PR #174** (תיקון ההרשאות המערכתי) ל-production — קריטי; בלעדיו משתמשים מוזמנים ייחסמו ברוב פעולות ה-E2E.
2. **4 משתמשי בדיקה**: admin, operator, vendor (עם פרופיל ספק מקושר!), agent/נציג.
3. **פרופיל ספק פעיל** עם is_active + availability_status=available + coverage/service מתאימים לשיוך.
4. **AgentShift פעיל** למוקדן (לבדיקת שיוך אוטומטי E2E-QUEUE-05).
5. **Twilio + Green API + VAPID** מוגדרים (אחרת בדיקות SMS/WhatsApp/Push מוגבלות ל-503/degraded).
6. **נתוני בדיקה**: קריאות במגוון סטטוסים; ≥2 ספקים תואמים (לבדיקת דחייה/הצעה מחדש); חוזה ספק שעומד לפוג (E2E-NOTIF-07).
7. **הרשאת מיקום בדפדפן** לבדיקות GPS.
8. **סביבה יציבה** — build תקין, ללא deploy במהלך הבדיקות.
