## 1. הנחיות כלליות לבודק

- כל מקרה בדיקה (Test Case) בנוי מ: **מזהה**, **תיאור/מטרה**, **צעדים**, **תוצאה צפויה**, ועמודת **תוצאה בפועל** לסימון ✅ עבר / ❌ נכשל / ⚠️ חלקי.
- מלאו את עמודת "תוצאה בפועל" תוך כדי הריצה, וצרפו **צילום מסך** לכל כשל.
- הבדיקה דורשת בדרך כלל **שני משתמשים במקביל** (מתפעל במחשב + ספק בנייד) — או משתמש אחד עם שני מסכים/דפדפנים.
- שימו לב לטקסטים בעברית ו-RTL: יישור לימין, כיווניות תקינה, אין חיתוך של טקסט.
- **זמן ריצה משוער למסע המרכזי (חלק א׳):** 40–60 דקות.
- כל שינוי סטטוס אמור להשתקף **תוך שניות** בכל המסכים הרלוונטיים (רשימת קריאות, תור, פורטל ספק, מפה, דשבורד) — זה חלק מהבדיקה, לא תופעת לוואי.

**מקרא רענון אוטומטי של מסכים (לצורך אימות סנכרון):**

| מסך | תדירות רענון |
|---|---|
| רשימת קריאות | 30 שניות |
| ניטור תור (QueueMonitor) | 15 שניות |
| התור שלי (MyQueue) | 30 שניות |
| פורטל ספק — קריאות | 30 שניות |
| פורטל ספק — בדיקת שיבוץ חדש | 10 שניות |
| צ׳אט בקריאה | ~3 שניות |
| GPS ספק (שליחת מיקום) | כל 30 שניות |
| מפת ספקים (VendorTracking) | 30 שניות |
| מפת כל הספקים (AllVendorsMap) | 90 שניות |
| התראות | מיידי (subscribe) |

## 2. הכנות לפני הבדיקה (Setup)

### 2.1 חשבונות בדיקה (חובה)
- [ ] משתמש **Admin (מנהל)** — סיסמה ידועה.
- [ ] משתמש **Operator (מוקדן/מתפעל)** — סיסמה ידועה.
- [ ] משתמש **Vendor (ספק)** — סיסמה ידועה, **משויך לרשומת ספק** (ישות `Vendor` קיימת ומקושרת למייל).
- [ ] (רשות) משתמש **Agent (טכנאי/נציג שטח)** — לבדיקת פורטל הטכנאי.
- [ ] "לקוח" עם **טלפון אמיתי** לקבלת SMS ומילוי משוב.

### 2.2 תשתיות והגדרות
- [ ] Secrets מוגדרים ב-Base44: `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_PHONE_NUMBER` (ל-SMS/WhatsApp), `GOOGLE_MAPS_API_KEY` (למרחק/ETA מדויק), מפתחות VAPID (ל-Push).
- [ ] הרשאות מיקום (Location) פתוחות בדפדפן/מכשיר של הספק.
- [ ] הרשאות התראות Push מאושרות בדפדפן.

### 2.3 מכשירים
- [ ] **מחשב** — עבור המתפעל/מנהל.
- [ ] **נייד אמיתי** — עבור הספק (חובה ל-GPS ולצילום תמונות).
- [ ] **טלפון "לקוח"** — לקבלת SMS ומילוי טופס משוב.

## 3. מפת מסכים, ישויות וקישוריות

### 3.1 הישות המרכזית וקשריה
`Call` (קריאה) היא **מקור האמת התפעולי**. סביבה שמסונכרנת אליה אוטומטית: `WorkQueue` (תור המתפעל) ו-`Case` (לצורכי דיווח).

```
Customer (לקוח) 1──*  Call (קריאה)  *──1 Vendor (ספק)
                          │
     ┌────────────────────┼──────────────────────────────┐
     ▼                    ▼                               ▼
 CallHistory        CallAssignmentAttempt            CallPhoto / Message
 (יומן שינויים)     (הצעת שיבוץ לספק)                (תמונות / צ׳אט)
     │                    │
 CallProduct / Deposit / EligibilityCheck / CallFeedback / Reminder

Call  ⇄  WorkQueue (queue_status)   ⇄  Case (status)   ← מסונכרנים אוטומטית
Vendor 1──* VendorLocation / VendorRating / VendorContract / VendorPayment / VendorPricing / VendorBreak
```

### 3.2 מסכים עיקריים לפי תפקיד

| תפקיד | מסכי מפתח |
|---|---|
| **מנהל/מתפעל** | Dashboard, Calls, CallDetails, NewCase, QueueMonitor, MyQueue, Customers/CustomerDetails, ServiceProviders/VendorDetails, AllVendorsMap, VendorTracking, Reports, HistoricalDataAnalysis |
| **ספק** | VendorPortal (דסקטופ + מובייל), VendorCallManagement, MyVendorProfile, VendorGuide |
| **טכנאי** | AgentDashboard, AgentCallManagement |
| **לקוח (ציבורי)** | CustomerPortal, CustomerFeedback (מבוסס טוקן, ללא כניסה) |

> **הערה על ניתוב:** הכתובת `/` מפנה לפי תפקיד — ספק ← `VendorPortal`, טכנאי ← `AgentDashboard`, אחרת ← `Dashboard`.

## 4. מכונת המצבים — סטטוסים של קריאה

טבלת הסטטוסים הקנונית (מתוך `src/lib/schemas/call.js` + תוויות `src/config/labels.js`):

| מפתח (key) | תווית בעברית | צבע |
|---|---|---|
| `waiting_treatment` | ממתין לטיפול | צהוב |
| `awaiting_assignment` | ממתין לשיבוץ | כתום |
| `assigning` | ספק שובץ | כחול |
| `vendor_enroute` | ספק בדרך | אינדיגו |
| `vendor_arrived` | נותן השירות הגיע | ענבר |
| `in_progress` | בטיפול | סגול |
| `cannot_complete` | לא ניתן לטפל | אדום |
| `in_storage` | באחסנה | אבן |
| `continued_treatment` | המשך טיפול | טורקיז |
| `awaiting_payment` | המתנה לחיוב | ורוד |
| `future_service` | שירות עתידי | סגלגל |
| `in_followup` | במעקב | תכלת |
| `completed` | סגור | ירוק |
| `cancelled` | בוטל | אפור |

**מעברי סטטוס מותרים לספק** (`updateVendorCall` — חובה שהמערכת תאכוף):
```
assigning / awaiting_assignment → vendor_enroute
vendor_enroute → vendor_arrived | in_progress | cannot_complete
vendor_arrived → in_progress | cannot_complete
in_progress   → completed | cannot_complete
```

**סנכרון אוטומטי** של סטטוס הקריאה לתור ולתיק (`Case`): כל שינוי ב-`Call.call_status` צריך לעדכן אוטומטית את `WorkQueue.queue_status` ואת `Case.status` (למשל `vendor_arrived → on_site`, `completed → completed`).

## 5. חלק א׳ — מסע הקריאה המרכזי מקצה-לקצה

> זהו התהליך המרכזי. הריצו אותו לפחות פעם אחת מלאה ("Happy Path") ופעם עם דחיית ספק ("שיבוץ מחדש").

### A1 — קליטת קריאה (4 ערוצים)

קיימים **4 ערוצי קליטה**. יש לבדוק לפחות את ערוץ המתפעל (A1.1). מומלץ לבדוק גם בוט/CTI אם מוגדרים.

| מזהה | תיאור | צעדים | תוצאה צפויה | תוצאה בפועל |
|---|---|---|---|---|
| **A1.1** | פתיחת קריאה ידנית ע״י מתפעל | כניסה כמתפעל → תפריט → "קריאה חדשה" (NewCase) → מילוי: שם לקוח*, טלפון לקוח*, כתובת איסוף* (שדות חובה), רכב, סוג שירות, עדיפות → "צור קריאה" | נוצרת קריאה עם מספר ייחודי (C-XXXXXXXX); סטטוס **"ממתין לטיפול"**; מעבר אוטומטי ל-`CallDetails`; נוצר פריט `WorkQueue`; נוצר `Case` תואם | |
| **A1.2** | ולידציה של שדות חובה | לנסות לשמור בלי שם/טלפון/כתובת | הודעות שגיאה בעברית: "שם לקוח הוא שדה חובה", "טלפון לקוח הוא שדה חובה", "כתובת איסוף היא שדה חובה"; הקריאה לא נוצרת | |
| **A1.3** | Geocoding של כתובת האיסוף | הזנת כתובת ישראלית אמיתית | הכתובת מומרת לקואורדינטות (משמשות לשיבוץ ספק לפי מרחק) | |
| **A1.4** | קליטה מבוט WhatsApp (99Digital) | (אם מוגדר) הזרמת קריאה דרך הבוט | נוצרת קריאה עם `created_by_source = bot`; שיבוץ אוטומטי למתפעל הפנוי; **הצעה אוטומטית לספק הטוב ביותר**; מוחזר מספר קריאה + קוד אימות ללקוח | |
| **A1.5** | זיהוי מתקשר ב-CTI/PBX | (אם מוגדר) שיחה נכנסת ממספר מוכר | חלון זיהוי (`CTICallerPopup`) מציג את הלקוח, מספר קריאות פתוחות ותאריך קריאה אחרונה; **לא** נוצרת קריאה עד שהמתפעל פותח ידנית | |

**אימות סנכרון לאחר A1:** הקריאה מופיעה ב-`Calls` (רשימה), ב-`QueueMonitor`, וב-`Dashboard`.

### A3 — בחירה ושיוך ספק

השיוך מבוסס **מנוע ניקוד** (מרחק 40, סוג שירות 20, דירוג 20, זמן תגובה 10, אחוז השלמה 10, בונוס סוג רכב 5, איזון עומס ±5) + מודל **הצעה→אישור** (הצעה בתוקף 10 דקות).

| מזהה | תיאור | צעדים | תוצאה צפויה | תוצאה בפועל |
|---|---|---|---|---|
| **A3.1** | פתיחת דיאלוג שיבוץ | ב-`CallDetails` → תפריט פעולות → "שבץ ספק" | נפתח דיאלוג "שיבוץ ספק לקריאה" עם בורר "סוג רכב שירות" (ניידת / גרר / גרר עם אחסנה) | |
| **A3.2** | שיבוץ אוטומטי לפי מיקום | לחיצה על "שיבוץ אוטומטי לפי מיקום" | מוצג הספק המומלץ: שם · מרחק בק״מ · הגעה ~X דק׳ · ציון; חישוב ETA דרך OSRM/Google Maps | |
| **A3.3** | סינון ספקים זמינים | פתיחת "או בחר ספק ידנית" | מוצגים **רק** ספקים `is_active` ו-`is_available_now`; ספקים עסוקים/לא-זמינים אינם מופיעים | |
| **A3.4** | ביצוע השיוך | בחירת הספק שלך → "שבץ" | סטטוס הקריאה משתנה ל-**"ספק שובץ" (assigning)**; `assigned_vendor_id` מתעדכן; נוצרת רשומת `CallAssignmentAttempt` (סטטוס `pending`, תוקף 10 דק׳); Toast: "הקריאה הוצעה ל-{ספק} — ממתין לאישור" | |
| **A3.5** | מניעת שיבוץ כפול | ניסיון לשבץ קריאה שכבר משובצת / כבר בטיפול | המערכת חוסמת (שגיאה 409): "כבר בטיפול" / "קיימת הצעה ממתינה" | |
| **A3.6** | החלפת ספק / קריאת המשך | על קריאה משובצת — לבחור "החלף ספק (אותה קריאה)" או "המשך עם ספק נוסף" | "החלף" → אותה קריאה, ספק חדש; "המשך" → נפתחת **קריאת המשך מקושרת** (`parent_call_id`) | |

### A4 — אישור/דחייה של הספק

| מזהה | תיאור | צעדים | תוצאה צפויה | תוצאה בפועל |
|---|---|---|---|---|
| **A4.1** | קבלת ההצעה בפורטל הספק | כניסה כספק בנייד → פורטל | מופיע חלון "קריאה חדשה!" עם **ספירה לאחור 120 שניות**, פרטי הקריאה, ותשלום משוער; כפתורים "קבל קריאה" / "דחה" | |
| **A4.2** | התראת Push לספק | להשאיר את הספק מחובר | התראת Push מגיעה תוך ~30 שניות (אם מופעל) | |
| **A4.3** | אישור קריאה | לחיצה "קבל קריאה" | הקריאה עוברת ל-**"ספק בדרך" (vendor_enroute)**; `CallAssignmentAttempt = accepted`; הספק עובר ל-`busy`; מתפעל מקבל התראה "ספק אישר קריאה" | |
